# Initial public preview release readiness

This document is the operator handoff for the first public binary release. Code
and workflows may establish the contract, but they must not fabricate external
signing, authorization, or qualification evidence.

## Implemented contract

- `0.0.z` is always beta and is always a GitHub prerelease.
- Release Please proposes patch increments only. Owners use the protected
  Manual Version Bump workflow for major or minor changes.
- Release Please never publishes or tags directly. The post-merge finalizer
  accepts only approved automated or manual release branches, revalidates the
  release contract, and pushes the tag with the owner token so tag CI runs.
- Linux ships x64 AppImage and zsync assets. Preview zsync metadata targets the
  exact immutable tag rather than GitHub's stable `latest` alias.
- Windows ships the intentionally unsigned x64 NSIS installer
  `GrokDesktop-beta-x64.exe`. It installs per-user without elevation and may
  trigger Unknown Publisher or Microsoft Defender SmartScreen warnings.
- Signed update manifests authorize exact artifact URL, size, and SHA-256.
  Linux atomically replaces the running AppImage. Windows downloads and
  revalidates the exact NSIS installer immediately before direct execution on
  explicit user confirmation. No shell or Authenticode trust is used.
- Publication emits release notes, signed update manifests, SPDX JSON SBOM,
  `SHA256SUMS`, GitHub attestations, and bounded release evidence.

## External prerequisites that must be supplied

1. Set repository secret `RELEASE_PLZ_TOKEN` to the owner-managed GitHub token
   used for protected release PRs. `OPENROUTER_API_KEY` stays optional.
2. Generate an Ed25519 preview update key. Put its private PEM only in
   `beta-release` as `GROK_UPDATE_SIGNING_PRIVATE_KEY_PEM`. Set key ID
   `preview-update-2026` and the matching DER/SPKI public-key JSON in
   `beta-build`, `beta-windows-build`, and `beta-release` as applicable.
3. Keep the qualified Windows toolchain paths, tested Windows ceiling, public
   trust values, and xAI evidence identifiers aligned with the disposable
   runner. Evidence identifiers must refer to real records for the pinned
   component digest. The Windows build needs no PFX, certificate password,
   SignTool path, signer thumbprint, or timestamp service.
4. Register a disposable Windows 11 x64 runner with labels `self-hosted`,
   `windows`, `x64`. Destroy it after the job, including the workspace, caches,
   and runner registration. An administrator verifies its
   live state separately with `gh api repos/grok-insider/grok-desktop/actions/runners`;
   the release automation token deliberately has no repository Administration
   permission.

  Windows SDK, Cargo cache, and Strawberry Perl paths. The isolated
  native builder also receives a canonical Visual C++ install root; this is the
  bounded marker `find-msvc-tools` requires before using the qualified MSVC
  executables on `PATH`.
- A disposable Windows 11 x64 runner still must be registered for each release
  candidate through the GitHub runner API as ephemeral with updates disabled.
  Its required default labels are `self-hosted`, `Windows`, and `X64`.
- The least-privilege `RELEASE_PLZ_TOKEN` is installed. Fine-grained personal
  access tokens are created and authorized in GitHub's web settings; `gh` can
  install a supplied value as a secret but cannot mint that credential.
- The manual `Release prerequisites` workflow validates every Windows core
  release input's schema without reading the protected values back through the
  GitHub API. Live self-hosted-runner inspection remains an administrator check
  so `RELEASE_PLZ_TOKEN` needs only Contents and Pull requests read/write.

## Promotion sequence

1. Merge the implementation PR into `dev`, let all required checks pass, then
   merge protected `dev` into `master`.
2. Run Release Preparation and review the next release PR. Do not merge it
   until all protected environment variables and secrets pass a dry preflight.
3. Merge the release PR to create the immutable candidate tag. Approve the
   Windows build only on the disposable runner.
4. Download the exact workflow artifacts before approving `beta-release`.
   Test the AppImage headlessly through Wisp/CDP and the unsigned NSIS installer
   on a clean Windows VM. Confirm the warning/bypass path, checksum and
   attestation, per-user install, application and daemon launch, deep link,
   update, and uninstall behavior.
5. Record hashes, OS image, test results, failures, and approver against the
   workflow run. Approve `beta-release` only if the exact artifacts pass.
6. Install the first published candidate and retain a clean update fixture. A
   later patch is not accepted until it proves an in-place manifest-authorized
   update from that fixture.

## Fail-closed conditions

No tag should be repaired manually. Missing Windows runner, update key, public
trust, xAI evidence, changelog entry, exact artifact QA, or promotion approval
stops publication. A Windows signing certificate is deliberately not a public
core prerequisite. Stable build and publication environments remain empty
until their update trust and release policy are qualified.
