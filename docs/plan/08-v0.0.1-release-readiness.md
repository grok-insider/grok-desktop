# v0.0.1 preview release readiness

This document is the operator handoff for the first public binary release. Code
and workflows may establish the contract, but they must not fabricate external
signing, authorization, or qualification evidence.

## Implemented contract

- `0.0.z` is always beta and is always a GitHub prerelease.
- Release Please proposes patch increments only. Owners use the protected
  Manual Version Bump workflow for major or minor changes.
- Release Please never publishes or tags directly. The post-merge finalizer
  accepts only approved automated or manual release branches, revalidates the
  release contract, and pushes the tag with the owner token so tag CI runs.
- Linux ships x64 AppImage and zsync assets. Preview zsync metadata targets the
  exact immutable tag rather than GitHub's stable `latest` alias.
- Windows ships x64 MSIX, exact-tag App Installer metadata, and the public
  preview certificate. It uses `GrokInsider.GrokDesktop.Preview` and
  `CN=Grok Desktop Preview`.
- Signed update manifests authorize exact artifact URL, size, and SHA-256.
  Linux atomically replaces the running AppImage. Windows downloads the exact
  MSIX, verifies Authenticode against packaged public trust, then opens the OS
  installer on explicit user confirmation. Electron Squirrel is not used.
- Publication emits release notes, signed update manifests, SPDX JSON SBOM,
  `SHA256SUMS`, GitHub attestations, and bounded release evidence.

## External prerequisites that must be supplied

1. Set repository secret `RELEASE_PLZ_TOKEN` to the owner-managed GitHub token
   used for protected release PRs. `OPENROUTER_API_KEY` stays optional.
2. Generate an Ed25519 preview update key. Put its private PEM only in
   `beta-release` as `GROK_UPDATE_SIGNING_PRIVATE_KEY_PEM`. Set key ID
   `preview-update-2026` and the matching DER/SPKI public-key JSON in
   `beta-build`, `beta-windows-signing`, and `beta-release` as applicable.
3. Generate the self-signed preview code-signing certificate with subject
   `CN=Grok Desktop Preview`. Put its base64 PFX and password only in
   `beta-windows-signing`; put its SHA-1 thumbprint in
   `GROK_WINDOWS_SIGNER_SHA1`. Do not commit either private input.
4. Populate the remaining documented Windows toolchain paths, exact store
   signing arguments, tested Windows ceiling, xAI provenance evidence ID, and
   xAI redistribution evidence ID. Evidence identifiers must refer to real
   records for the pinned component digest.
5. Register a disposable Windows 11 x64 runner with labels `self-hosted`,
   `windows`, `x64`. Destroy it after the job, including certificate store,
   workspace, caches, and runner registration.

## Promotion sequence

1. Merge the implementation PR into `dev`, let all required checks pass, then
   merge protected `dev` into `master`.
2. Run Release Preparation and review the `v0.0.1` release PR. Do not merge it
   until all protected environment variables and secrets pass a dry preflight.
3. Merge the release PR to create `v0.0.1`. Approve Windows signing only on the
   disposable runner.
4. Download the exact workflow artifacts before approving `beta-release`.
   Test the AppImage headlessly through Wisp/CDP and the MSIX on a clean Windows
   VM after verifying the published certificate fingerprint.
5. Record hashes, OS image, test results, failures, and approver against the
   workflow run. Approve `beta-release` only if the exact artifacts pass.
6. Install `v0.0.1` and retain a clean update fixture. A later `v0.0.2` is not
   accepted until it proves an in-place signed update from that fixture.

## Fail-closed conditions

No tag should be repaired manually. Missing Windows runner, PFX, update key,
public trust, xAI evidence, changelog entry, exact artifact QA, or promotion
approval stops publication. Stable environments stay intentionally empty until
a publicly trusted stable certificate and stable package identity are ready.
